# ADR-048 Review: Project-Level Tiered Context System

## Verdict: APPROVE_WITH_CHANGES

Az ADR-048 **alapvetően szilárd javaslat**, amely természetes kiterjesztése a meglévő tiered memory rendszernek. A tökenbankáló probléma (~10.8k → ~500 token) megoldása fontos, és a hierarchikus struktúra logikus.

**Azonban 3 kritikus pontot kell tisztázni/javítani mielőtt implementálásra kerül.**

---

## Erősségek

### 1. Architektúra illeszkedés ✅

**Konzisztencia a memoryStore.ts-sel:**
- ✅ Tier hierarchy (hot/warm/cold/shared) pontosan ugyanaz
- ✅ SQLite FTS5 alapú indexing — már bevált pattern
- ✅ Salience + decay mechanizmus megegyezik
- ✅ Terminal scoping kompatibilis

**Kód szinten:**
```typescript
// Meglévő (memoryStore.ts:97-126)
export const TIER_POLICIES: Record<MemoryTier, TierPolicy> = {
  hot:   { tier: 'hot',   maxAge: '48h',    decayRate: 0.15 },

  cold:  { tier: 'cold',  maxAge: '365d',   decayRate: 0.01 },

// ADR-048 javaslat (line 46-53) — UGYANAZ ✅
| hot    | 48h  | 15%/nap |
| warm   | 14d  | 5%/nap  |
| cold   | 365d | 1%/nap  |
| shared | ∞    | 0%      |
```

**Erősség:** Nincs architektúra friction, drop-in extension lehet.

### 2. Tier Policy — TTL és Decay ✅

**TTL értékek helyesek:**
- ✅ **hot (48h):** Megfelelő aktuális munkához — 2 napon belüli DONE-ok relevánsak
- ✅ **warm (14d):** Sprint ciklus (1-2 hét) lefedése
- ✅ **cold (365d):** Architekturális döntések (ADR-ek) 1 évig frissek

**Decay rate-ek jól kalibáltak:**
- ✅ **hot (15%/nap):** Agresszív — 7 nap után ~68% salience loss → értelmes
- ✅ **warm (5%/nap):** Mérsékelt — 14 nap után ~49% salience loss
- ✅ **cold (1%/nap):** Lassú — 365 nap után sem 0, de fokozatosan csökken

**Math check:**
```
hot:  7 nap után = 0.85^7 ≈ 0.32 (68% loss) ✅
warm: 14 nap után = 0.95^14 ≈ 0.51 (49% loss) ✅
cold: 90 nap után = 0.99^90 ≈ 0.41 (59% loss) ✅
```

**Erősség:** Decay rate-ek matematikailag konzisztensek a memoryStore.ts implementációval.

```
TERMÉK (SpaceOS)
  └── shared: vízió, 5 Golden Rule
```

## Változtatási javaslatok

### 🔴 CRITICAL #1: SQLite séma UNIQUE constraint túl szűk

**Probléma:**
```sql
-- ADR-048 javaslat (line 94)
UNIQUE(product, project, tier)
```

Ez azt jelenti, hogy **egy projektnek tier-enként csak EGY context entry-je lehet**. De mi van ha:
- Több epic is aktív ugyanazon a projekten (cutting: epic-1 + epic-2)?
- Több blocker is van ugyanabban a tier-ben?

**Javaslat:**
```sql
-- Távolítsd el a UNIQUE constraint-et
-- Helyette: több entry tier-enként, de tag/source alapján szűrve

CREATE TABLE IF NOT EXISTS project_context (
  id INTEGER PRIMARY KEY,
  product TEXT NOT NULL,
  project TEXT NOT NULL,
  tier TEXT NOT NULL,
  source TEXT NOT NULL,           -- 'epic', 'blocker', 'decision', 'test_summary'
  summary TEXT NOT NULL,
  metadata TEXT,
  salience REAL DEFAULT 0.5,
  created_at TEXT DEFAULT CURRENT_TIMESTAMP,
  updated_at TEXT DEFAULT CURRENT_TIMESTAMP,
  expires_at TEXT,
  UNIQUE(product, project, tier, source)  -- ← több source lehet tier-enként
);
```

**Rationale:** Flexibilis modell, nem szorul be egy monolitikus summary-ba.

### 🟡 MEDIUM #2: Tömörítési stratégia — cache invalidation nincs specifikálva

**Hiány:**
Az ADR-048 említi hogy "Cache invalidation logika helyes" (line 203), de **nincs definiálva mi triggerel cache invalidation-t**.

**Példa forgatókönyv:**
1. Backend terminal DONE-ol egy taskot (cutting projekt, hot tier)
2. Pipeline futtatja a Haiku tömörítést
3. De mi van ha 5 perc múlva újabb DONE érkezik?
   - Újra tömörít? (költség: ~500 token Haiku call)
   - Vagy várja a következő batch-et? (staleness: 5 perc)

**Javaslat:**
Adj hozzá egy **Cache Policy** szekciót:

```markdown
### Cache Policy

| Tier | Invalidation Trigger | Max Staleness |
|------|----------------------|---------------|
| hot  | DONE/BLOCKED outbox  | 5 minutes     |
| warm | Napi digest          | 24 hours      |
| cold | ADR merge, heti retrospective | 7 days |
| shared | Manual refresh (Vision frissítés) | ∞ |

**Batching:** Hot tier változások 5 percig batch-elve → 1× Haiku call / 5 perc.
```

**Rationale:** Token cost optimization + staleness trade-off explicit.

### 🟡 MEDIUM #3: Pipeline integráció — ADR merge hook hiányzik

**Probléma:**
Az ADR-048 azt írja (line 139):
```
ADR merged → cold tier frissül
```

De **nincs ADR merge hook a pipeline-ban** jelenleg. A git hook-ok csak commit-ra futnak, nem merge-re.

**Javaslat:**
Pontosítsd hogy **hogyan** detektálja a pipeline az ADR merge-t:

**Opció A: Post-merge hook** (ajánlott)
```bash
# .git/hooks/post-merge
if git diff-tree --name-only HEAD@{1} HEAD | grep -q "^docs/architecture/decisions/ADR-.*\.md$"; then
  echo "ADR merged, triggering cold tier refresh"
  curl -X POST localhost:3456/api/pipeline/refresh-cold-tier
fi
```

**Opció B: Napi scan** (egyszerűbb, de lassabb)
```typescript
// nightwatch.sh hourlyDigest után
// Napi egyszer: git log --since="24 hours ago" --grep="ADR-.*" → cold tier refresh
```

**Rationale:** Implementációs részleteket kell tisztázni.

## Kérdések / Tisztázandók

### ❓ Q1: Shared tier tömörítés — mennyi ideig cache-el?

A Vision file (~8k token) → shared tier summary (~500 token) tömörítés **drága**:
- Input: 8000 token
- Output: 500 token
- Haiku költség: ~0.25¢ / tömörítés

De a Vision file **ritkán változik** (hónapokra statikus).

**Kérdés:** Mennyi ideig cache-eljük a shared tier summary-t?
- Option A: 30 nap (ajánlott — Vision nem változik gyakran)
- Option B: ∞ (manual invalidation only)
- Option C: 90 nap

**Javaslat:** Adj hozzá egy `shared_tier_cache_ttl` config-ot.

---

### ❓ Q2: Multi-product támogatás — SpaceOS vs Datahaven

Az ADR-048 példa (line 59):
```typescript
product: string; // "spaceos" | "datahaven"
```

De a jelenlegi EPICS.yaml **nincs product scope-olva** — minden epic "spaceos".

**Kérdés:** Tervezed a Datahaven-t külön product-ként kezelni?
- Ha igen → EPICS.yaml frissítés kell (product mező hozzáadása)
- Ha nem → akkor a product mindig "spaceos"?

**Javaslat:** Pontosítsd a product scope-ot vagy hagyd opcionálisnak:
```typescript
product?: string; // Default: "spaceos"
```

---

### ❓ Q3: Test counts — minden projekthez releváns?

A `testCounts` mező (line 68-71) csak backend/frontend projektekhez értelmes:
```typescript
testCounts?: {
  backend: number;
  frontend: number;
}
```

De mi van a **Datahaven Nexus** projekttel (nincs backend/frontend, csak Node.js)?
- Vagy **Sales** modul (csak .NET backend)?

**Javaslat:** Flexibilisebb metadata struktúra:
```typescript
testCounts?: Record<string, number>; // { "backend": 994, "frontend": 941, "e2e": 12 }
```
