# Observation Report - MCP Development Opportunities (2026-02-26-004)

## Mi történt?
Elemeztem a `McpService`, `McpController` és a kapcsolódó ágens-feladat kezelő (`AgentTasksController`) kódját, hogy feltárjam az ágens-kommunikáció javításának lehetőségeit.

## Mit láttam? (Facts)
- **Kölcsönhatás módja:** A rendszer SSE-t használ a kapcsolódáshoz és szabványos HTTP POST-ot az üzenetküldéshez. Ez egy modern, stabil alap.
- **Állapotmentesség:** Az MCP szerver jelenleg teljesen állapotmentes az üzenetek tekintetében. Nincs naplózva, hogy melyik ágens milyen eszközt hívott meg, és mi volt a válasz.
- **Szinkronitás:** Bár a háttérben vannak aszinkron hívások, a belső `IMcpService` interfész és a kontroller közötti kapcsolat blokkoló jellegű (`.GetResult()`), ami skálázhatósági korlátot jelenthet.
- **Híd az entitásokhoz:** Az `AgentTasksController` már képes az `AgentTask` entitások kezelésére, de ez a folyamat jelenleg el van szigetelve az MCP üzenetkezeléstől.

## Milyen fejlesztési lehetőségeket (Opportunities) látok?

### 1. Központi Üzenetnapló (Persistent Message Log)
- **Leírás:** Minden bejövő `JsonRpcRequest` és kimenő `JsonRpcResponse` mentése az adatbázisba (`flow.db`).
- **Előny:** Visszakövethető, pontosan mit csinált az ágens. A React UI-on megmutatható a "Brain log".

### 2. Aszinkron Eseménykezelés (Reactive MCP)
- **Leírás:** Az SSE csatorna kihasználása szerver-oldali push értesítésekre. Ha az adatbázisban változik egy projekt állapota, a szerver azonnal értesítheti az érintett ágenst.
- **Előny:** Gyorsabb reakcióidő, nincs szükség folyamatos polingra az ágensek részéről.

### 3. "Serviceable" Üzenetsémák
- **Leírás:** Olyan speciális MCP eszközök bevezetése, amelyek segítik az ágensek közötti üzenetváltást (pl. `send_handover_report`).
- **Előny:** Egységesített formátum az ágensek közötti váltáshoz (Handoff), ami csökkenti az információvesztést.

### 4. Hierarchikus Telemetria
- **Leírás:** Az MCP hívások összekapcsolása a hierarchia szintekkel (pl. melyik Epic-hez tartozik a hívás).
- **Előny:** Láthatóvá válik, hogy melyik mérföldkőnél mennyi "ágens-munka" (token/idő) keletkezett.

## Javaslat (Explorer):
Az MCP szervert érdemes lenne felruházni egy "Audit Trail" képességgel. Ez az első lépés afelé, hogy az ágensek ne "sötétben" dolgozzanak, hanem a rendszer lássa és kezelje minden lépésüket. Ez közvetlenül támogatja a felhasználó igényét a folyamatok jobb nyomonkövethetőségére.
