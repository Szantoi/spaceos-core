# TASK-18-04: ChromaDB highlights indexing

## Goal

Create and maintain a dedicated `episode_highlights` ChromaDB collection for semantic highlight retrieval.

## Affected Files

- `src/rag/episodicMemory.ts` — add highlights collection methods
- `src/metadata/qualityScoring.ts` — similarity input helpers
- `src/tests/unit/highlights-chromadb.test.ts` — NEW

## Acceptance Criteria

Given stored highlights,
When highlight indexing sync runs,
Then:

- AC-1: Collection `episode_highlights` is created or reused.
- AC-2: Document embedding content combines decisions + lessons + next steps.
- AC-3: `search_highlights(query, topK)` returns semantically relevant highlights.
- AC-4: Sync metadata is persisted in `highlights_chromadb_sync`.
- AC-5: Missing ChromaDB gracefully degrades with explicit warning.
- AC-6: Unit tests are green.

## Implementation Status

- Semantic highlight index implemented in `src/rag/episodicMemory.ts` with dedicated collection name `episode_highlights`.
- Sync metadata persisted through `highlights_chromadb_sync` and deterministic `highlight:{id}` vector ids.
- Runtime Vitest verification is still pending in this environment.

## Implementation Hints

- Reuse existing EPIC-12 vector store adapter contracts.
- Keep collection naming constant: `episode_highlights`.
- Store `highlight_id`, `session_id`, `domain`, `track` in metadata.

## Risks

- R1: Embedding model mismatch between episodes and highlights.
  - Mitigation: explicitly store model id per vector row.
- R2: Duplicate vectors under retries.
  - Mitigation: deterministic vector id based on `highlight_id`.
