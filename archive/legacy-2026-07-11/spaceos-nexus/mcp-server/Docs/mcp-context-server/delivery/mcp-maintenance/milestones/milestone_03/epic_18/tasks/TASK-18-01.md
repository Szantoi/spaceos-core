# TASK-18-01: Episode highlights schema migration

## Goal

Create the database foundation for self-reflection by adding `episode_highlights`, `highlight_feedback`, and `highlights_chromadb_sync` tables with idempotent migration behavior.

## Affected Files

- `src/metadata/migrations/009_epic18_highlights_schema.sql` — NEW
- `src/mcp/AgentDb.ts` — load migration and add read/write helpers
- `src/tests/unit/episode-highlights-schema.test.ts` — NEW

## Acceptance Criteria

Given a fresh or existing `agent.db`,
When schema initialization runs,
Then:

- AC-1: `episode_highlights` table exists with columns: `id`, `episode_id` (UNIQUE), `key_decisions`, `lessons`, `next_steps`, `quality_score`, `ai_generated`, `ai_model`, `ai_tokens_used`, `created_at`.
- AC-2: `highlight_feedback` table exists with FK to `episode_highlights(id)`.
- AC-3: `highlights_chromadb_sync` table exists and tracks vector sync metadata.
- AC-4: Migration is idempotent; repeated init does not fail.
- AC-5: FK integrity works: deleting highlight cascades feedback.
- AC-6: Unit tests are green.

## Definition of Done

- [x] Migration file created and loaded by schema initialization
- [x] AgentDb helper methods for highlights table access added
- [x] Unit tests cover schema + idempotency + FK behavior
- [x] No TypeScript errors
- [ ] Code review approved

## Implementation Hints

- Follow `008_epic17_domain_schema.sql` pattern for safe idempotent DDL.
- Store arrays as JSON text columns (`TEXT`) for compatibility.
- Keep `quality_score` nullable initially for backward-safe phased rollout.

## Risks

- R1: FK mismatch with existing `episodes` schema variant.
  - Mitigation: introspect `episodes` PK type in tests before inserting fixtures.
- R2: Migration ordering conflict with prior epics.
  - Mitigation: append as `009_...` and only load once from a single initializer path.
