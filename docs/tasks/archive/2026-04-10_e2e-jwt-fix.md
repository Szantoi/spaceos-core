# E2E JWT autentikációs hiba — Diagnosztika és javítás

**Dátum:** 2026-04-10 – 2026-04-11
**Kontextus:** E2E tesztek futtatása (`npm test`) 401 hibával megáll a global-setup chain ellenőrzésnél.

---

## 1. Probléma

Az E2E tesztek nem futnak le — a `global-setup.ts` BFF→Kernel chain check 401-et kap:

```
❌ BFF→Kernel chain broken: GET /bff/api/tenants returned 401
```

**Kiváltó okok (több rétegű):**
1. Keycloak `/auth/` path prefix — a realm endpointok `/auth/realms/spaceos/...` alatt vannak
2. Issuer mismatch — token: `https://joinerytech.hu/auth/realms/spaceos`, config: `http://localhost:8080/realms/spaceos`
3. Audience mismatch — token: `account`, config: `kernel-api`
4. Published Kernel elavult — nem tartalmazta a Keycloak Authority konfigurációt
5. Role mapping hiányzott a tokenből

---

## 2. Felfedezett architektúra

### 2.1 Keycloak path struktúra

A VPS-en futó Keycloak a `/auth/` prefix-et használja:
```
http://localhost:8080/auth/realms/spaceos/...              ← működő path
http://localhost:8080/realms/spaceos/...                   ← 404
```

### 2.2 OIDC discovery (helyes)
```json
{
  "issuer": "https://joinerytech.hu/auth/realms/spaceos",
  "jwks_uri": "http://localhost:8080/auth/realms/spaceos/protocol/openid-connect/certs"
}
```

### 2.3 Service menedzsment
- **Kernel**: systemd (`spaceos-kernel.service`), env: `/etc/spaceos/kernel.env`, port 5000
- **Orchestrator**: PM2 (`spaceos-orchestrator`), env: dotenv `.env`, port 3000
- **Keycloak**: port 8080, `/auth/` prefix, frontend URL: `joinerytech.hu`

### 2.4 Authorization policy-k (Kernel Program.cs:247-254)
```
ReadPolicy  → RequireRole("Joiner", "Designer", "Admin")
WritePolicy → RequireRole("Designer", "Admin")
AdminPolicy → RequireRole("Admin")
```

## 3. Elvégzett javítások

### 3.1 E2E `.env` — KC_URL fix
```diff
- KC_URL=http://localhost:8080
+ KC_URL=http://localhost:8080/auth
```
**Státusz:** KÉSZ

### 3.2 Orchestrator `.env` — issuer, audience, JWKS fix
```diff
- JWKS_URI=http://localhost:8080/realms/spaceos/protocol/openid-connect/certs
- JWT_ISSUER=http://localhost:8080/realms/spaceos
- JWT_AUDIENCE=kernel-api
+ JWKS_URI=http://localhost:8080/auth/realms/spaceos/protocol/openid-connect/certs
+ JWT_ISSUER=https://joinerytech.hu/auth/realms/spaceos
+ JWT_AUDIENCE=kernel-api
```
**Státusz:** KÉSZ, PM2 restart (`pm2 restart spaceos-orchestrator`)

### 3.3 Kernel appsettings — Authority `/auth/` prefix
**Fájlok módosítva:**
- `SpaceOS.Kerner/SpaceOS.Kernel.Api/appsettings.Development.json`
- `SpaceOS.Kerner/SpaceOS.Kernel.Api/appsettings.Production.json`

```diff
- "Authority": "http://localhost:8080/realms/spaceos"
+ "Authority": "http://localhost:8080/auth/realms/spaceos"
```
**Státusz:** KÉSZ

### 3.4 Kernel újra-publish
```bash
dotnet publish SpaceOS.Kernel.Api -c Release -o /opt/spaceos/spaceos-kernel/publish
systemctl restart spaceos-kernel
```
**Döntés:** Publish a forrásból, nem kézi config patch a publish mappában — stabilabb.
**Státusz:** KÉSZ

### 3.5 Keycloak: audience mapper hozzáadva
A `test-runner` clienthez `kernel-api` custom audience mapper hozzáadva Admin API-n keresztül.
Token `aud` értéke ezután: `["kernel-api", "account"]`
**Státusz:** KÉSZ

### 3.6 Keycloak: Admin role mapping ellenőrzés
A `test-admin` user már rendelkezett `Admin` realm role-lal. A `realm_access.roles` tartalmazza a tokenben, a Kernel `OnTokenValidated` event mappeli `ClaimTypes.Role`-ra.
**Státusz:** OK, nem kellett változtatni
