## ⚠️ ÉSZAKI CSILLAG — domén-független, többágazatú architektúra (verticalizálhatóság)

> **Cél (2026-06-15, felhasználói irány):** a UI-t ÉS az egész rendszert úgy kell modulokra vágni, hogy **más doménekre is használható** legyen — ne csak asztalos-/faipari gyártásra. Például egy **pékség** üzemeltetésére (receptúra/BOM, sarzs-gyártás, lejárat/HACCP, napi sütés-ütemezés, bolti eladás) ugyanazokkal a magmodulokkal kell menjen. Ez NEM azonnali refaktor, hanem **tervezési elv**: minden új modulnál tartsd szem előtt, és ahol olcsó, már most a domén-független irányba told.

**A szétválasztás elve — három réteg:**
1. **Domén-független MAG (újrahasznosítható minden ágazatra):** FSM-motor + státusz-governance · CRM/lead-pipeline · ajánlat→rendelés→számla pénzügyi gerinc · katalógus/cikkszám-törzs + jóváhagyás · raktár (lot/zóna/mozgás) · beszerzés (RFQ→PO→bevét) · partner/beszállítói portál · HR/jelenlét · kontrolling (EAC, óradíj-bázis) · dokumentumtár (DMS) · EHS · feladat-aggregátor (`unifiedTasks`) · vezetői BI-cockpit · AI munkaterület. Ezek a **láncszem-primitívek**, nem asztalos-specifikusak.
2. **Domén-ADAPTER (ágazatonként cserélhető):** a szakma-specifikus tudás és folyamat — asztalosnál a `woodwork_domain.md` (tömörfa vs. lapanyag útvonalak, nesting/szabászat, élzárás, BOM-sablonok); pékségnél ennek megfelelője lenne (receptúra-skálázás, kovászos/sarzs-útvonal, sütés-ütemezés, eltarthatóság/HACCP). A magmodulok ezt **adatként/konfigurációként** fogyasztják, nem hardcode-olva.
3. **MÁRKA/megjelenés (ügyfelenként cserélhető):** `branding` store (küldetés/szín/hangnem/persona) + akcentus-paletta — már ma is külön rétegben él.

**Gyakorlati következmények új munkánál (tartsd be):**
- A domén-specifikus szótárt/folyamatot ne süsd bele a magmodulba — vezesd **adapter-dokumentumba** (asztalos: `woodwork_domain.md`), a magmodul paraméterezhető legyen.
- Kerüld a hardcode-olt asztalos-fogalmakat a mag-UI-ban (pl. „szabászat", „élzárás" mint fix címke) — ahol megoldható, a `PROD_KINDS`/művelettípus-szerű **konfigból** jöjjön, hogy egy pékség-adapter más műveleteket adhasson.
- A státusz-FSM-ek, a `unifiedTasks`, a kontrolling-motor, a portál-minták MARADJANAK domén-semlegesek.
- Ez ma **vezérelv**, nem migráció: ha egy döntésnél két út közül az egyik domén-függetlenebb és nem drágább, azt válaszd. Külön „domén-absztrakciós" epiket a 6.5 stratégiai blokk tartalmaz (PROJECT_STATUS.md).

> A fejlesztést EZEK MENTÉN végezzük: ezek azok a folyamatok, amelyeken a **vevő bekerül**, és a végén **terméket + élményt** kap. Minden új funkciót pozícionálj egy értéklánc láncszemeként. (Forrás-tudás: USER_PROFILES.md vevő-típusok; woodwork_domain.md a dokumentum-lánc.)

1. **Egyedi / projekt bútor — FŐ lánc** (B2B gyártó A1–A4, belsőépítész B1–B3, magánszemély D1–D2):
   CRM (lead → lehetőség) → **Belsőépítészet/Tervezés** (koncepció, **látványterv**, Konfigurátor, Bútorsor) → **Árajánlat** (Értékesítés) → Megrendelés + **Szerződés** (Pénzügy) → **Gyártás-előkészítés** (anyag/útvonal/dokumentum/**kiadás a műhelynek**) → **Gyártás** (Műhely-terminál + Üzemvezető) → **Minőség** (QA) → **Logisztika** (kiszállítás + helyszíni beépítés/átadás) → **Pénzügy** (szállítólevél → számla) → **Reklamáció/Service** (jótállás). *Élmény:* látványterv, **munkaszám/QR-követés**, beépítés, átadás, garancia.

   Webshop böngészés → **Konfigurátor** („Összeállítom") VAGY kosár → rendelés a közös pipeline-ba (draft) → Gyártás VAGY raktári készlet → Logisztika → Pénzügy. Webshop-érdeklődés → **auto-lead** a CRM-be. *Élmény:* önkiszolgáló konfigurálás, ársáv, rendelés-követés.

4. **B2B platform-bérmunka** (cégek közti lánc):
   Gyártó → **delegálás** (epic / bérmunka / CRM-lehetőség) másik platform-cégnek (**B2BHandshake**: „kiadva" ↔ „beérkezett megbízás") → partner gyártja → vissza. Ide tartozik a belső-egység `internal_order` kézfogás is.

## ⚠️ STÁTUSZOK — a legfontosabb szabály

A rendszer **állapotgépekre (FSM)** épül. Minden entitásnak meghatározott státuszai és **engedélyezett átmenetei** vannak. **Soha ne találj ki új státuszt** és ne ugorj át lépést — mindig a meglévő láncot használd, és az átmenetet a store megfelelő akcióján keresztül végezd, hogy átgyűrűzzön a többi modulba.

### Cikkszám / katalógus-tétel életciklus (`catalog[].status`) — a TÖRZSADAT világ a governance-otthon
`draft → review → active` fő-ág; `incomplete` (hiánypótlás), `rejected` (elutasítva), `archived`. A cikkszám-törzs **kereszt-metsző governance-adat**: a kanonikus otthona a **Törzsadat** világ (`masterdata`, `page-masterdata.jsx` — Cikkszámok + Jóváhagyások), de a Tervezés / Beszerzés / Raktár / Értékesítés mind **fogyasztó**. A státusz a **tételen** él (világ-független), nem egy világ tulajdona.
- **A kulcs-szabály:** bárki felvehet **piszkozatot** a MINIMUM mezőkkel (Megnevezés + Kategória + Típus; auto cikkszám) — `addCatalogDraft(data)`. A draft/incomplete/review tétel **belül használható** (hivatkozható, **igényelhető**, bevételezhető, tervezhető vele), de **ajánlatba / eladásba / webshopba NEM** kerülhet, amíg jóvá nem hagyják.
- **Jóváhagyáshoz (active) kötelező** (`catalogCompleteness(item)` → `{checks, ready, missing}`): Eladási ár · Beszállítói forrás/beszerzési ár (procCatalog-forrás vagy saját `suppliers` ár) · kategória műszaki paraméterei (a `categoryFields` number/select/material mezői) · Láthatóság · Megnevezés+Kategória+Típus. Amíg hiányzik → a „Beküldés" / „Jóváhagyás" gomb **LEZÁRT** (disabled + tooltip a hiányzó mezőkkel), nem rejtett.
- **Engedélyezett átmenetek** (`_catFlow`, `catStatusCanGo(from,to)`): `draft→[review,archived]`, `incomplete→[review,archived]`, `review→[active,incomplete,rejected]`, `rejected→[draft,archived]`, `active→[archived,review]`, `archived→[draft]`. Tiltott = LEZÁRT gomb.
- **Átmenet:** **`setCatalogStatus(id, to, {reason})`** — validált FSM. `active`/`incomplete`/`rejected` **`catalog.approve` joghoz** kötött; `incomplete`/`rejected`-nél **indok kötelező**. `active`-ra váltáskor teljesség-ellenőrzés. Lekérdezés: `catalogByStatus(status)`.
- **Eladhatóság-kapu:** **`isCatalogSellable(item)`** + **`sellableCatalog()`** (csak `active`). A fogyasztói felületek EZT használják a nyers `sim.catalog` helyett: **ajánlat ItemBuilder** (`page-sales.jsx`, `page-sales-detail.jsx` — `...sim.sellableCatalog()`), **webshop** (`shopProducts()`/`shopCatalogItems()` — `status==="active"` szűr). A **Kereskedelem pultos eladás** külön `tradeProducts` törzsből dolgozik (nem `sim.catalog`), ezért rá a kapu nem hat. A **beszerzési igény** ItemBuilder NEM gated (a beszerzés belépési pont — draft tétel is igényelhető).
- **Jog:** új `catalog.approve` (külön jóváhagyó szerep, `portal.jsx` + `acc-internal`). **LS_KEY `jt_sim_v47`, seed `v: 30`** (új: `status`/`statusReason`/`createdBy`/`approvedBy` a tételen, `catDraftSeq`; demó draft/incomplete/review tételek: `c-draft-led`, `c-inc-fog`, `c-rev-fog`).

### Beszállítói cikk-megfeleltetés — `page-suppliermap.jsx`

- **Megőrzés:** `localStorage` kulcs `jt_sim_vN`. **Séma-változáskor EMELD a verziót** (`LS_KEY` + a `load()`/`seed()` `v` mező), különben a régi mentés ütközik.
- **Demó visszaállítás:** `window.sim.reset()` (Tweaks panel „Demó visszaállítása").
- Olvasás komponensben: `const s = useSim();`.

## Technikai konvenciók
- **Babel-scope:** minden `<script type="text/babel">` külön scope. Megosztott komponenst exportálj `Object.assign(window, {...})`-szal a fájl végén; más fájl `window`-ról éri el.
- **Stílusobjektum-névütközés TILOS** — egyedi nevű `const xStyles = {...}` vagy inline; soha `const styles = {...}`.
- **Betöltési sorrend számít** (`JoineryTech Portal.html`): előbb `data-*.js`, aztán `*.jsx`. Új store-mező a seedben → a `data-*` globálisok már elérhetők.
- **Stílus:** Tailwind utility; paletta **stone** + **teal**, modulonként akcent-szín. Magyar UI-szövegek.
- **Mobil:** min. 44px érintési cél; nincs vízszintes kilógás; alulról nyíló panelek `env(safe-area-inset-bottom)` térközzel; részlet-panel (SlideOver) az alsó menü FÖLÉ (z-50).
- **Táblázat-minta:** asztali táblázat = mobilon tömör koppintható sor → részlet-panel (ne zsúfolt mini-táblázat).

## ⚠️ React Hooks — kritikus szabályok (megsértve = végtelen loop / crash)

### 1. Korai return ELŐTT minden hook
`useState`/`useMemo`/`useEffect` hívások MINDIG a komponens elejére kerüljenek, **soha** feltételes return után. Ha egy komponensnek pl. egy tab-váltásnál különböző JSX-t kell visszaadnia, a switch/conditional a hookOK UTÁN jön:
```jsx
// ✅ HELYES
function MyComp({ tab }) {
  const sim = useSim();              // 1. hook
  const [q, setQ] = useState("");    // 2. hook
  const data = useMemo(...);         // 3. hook
  if (tab === "other") return <OtherView />;  // korai return UTOLJÁRA
  return <MainView />;
}
// ❌ HIBÁS — végtelen loop / "Rendered fewer hooks than expected"
function MyComp({ tab }) {
  const sim = useSim();
  if (tab === "other") return <OtherView />;  // ← hook ELŐTT return → TILTOTT
  const [q, setQ] = useState("");   // ← ezt React már nem hívja meg
}
```

### 2. `getState()` memoizálás (`useSyncExternalStore`)
`api.getState()` **mindig ugyanazt az objektumreferenciát** adja vissza, amíg `state` nem változik. Ha új objektumot adsz vissza (`{ ...state }`) minden hívásnál, végtelen re-render keletkezik. **A derived objektum az api-METÓDUSOKAT IS tartalmazza** (`{ ...api, ...state, derivedField }`), így a `const sim = useSim(); sim.valamiMetodus()` minta is működik (nemcsak `sim.valamiAdat`). A metódusok stabil referenciák, a derived objektum csak state-változáskor épül újra → memoizálás-biztos. Pattern:
```js
getState() {
  if (api._cachedRaw === state) return api._derivedState;  // ← stable ref
  api._cachedRaw = state;
  api._derivedState = { ...api, ...state, derivedField };   // ← api-metódusok + state
  return api._derivedState;
}
```
> ⚠️ Történeti hiba: amíg a derived objektum CSAK a state-et tartalmazta, a `const sim = useSim(); sim.hasPerm()` / `sim.setCatalogStatus()` minta csendben elhasalt (`… is not a function`) — pl. a Törzsadat → Jóváhagyások és a katalógus-szerkesztő státusz-szekciója crashelt. A fenti `...api` merge a gyökér-ok javítás.

### 3. `set()` nem hívható render közben
A `set(patch)` → `emit()` → `listeners` lánc szinkron, azonnal re-rendert vált ki. Ha render közben hívod (pl. egy useMemo-ban), végtelen loop. **Csak event handler-ből vagy `useEffect`-ből.**

### 4. Babel fordítási idő
Nagy fájlok (>500 sor) esetén a Babel in-browser fordítás lassú — az iframe 10s-en belül nem tölt be, de ez **nem hiba**, csak lassúság. Ha `done` "WARNING: iframe did not finish loading" üzenetet ad de nincs console error, a lap valójában betölt. Ellenőrzés: `screenshot_user_view` vagy hosszabb `sleep` után `eval_js`.
