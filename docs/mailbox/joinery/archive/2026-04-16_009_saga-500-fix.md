# MSG-JOINERY-009 — Saga 500 kivizsgálás: POST /bff/door-orders

## Probléma

Az E2E-033 futása során a `43-joinery-cutting-integration` teszt `POST /bff/door-orders`-re
500-as választ kapott. Ugyanez a route a `29-joinery-order` tesztben zöld — tehát
valamilyen állapot- vagy sorrend-függő hiba van.

## Feladat

### 1. Diagnózis

A Joinery modul legfrissebb deployja (INFRA-101, 2026-04-16) óta keletkezett-e regresszió?
Ellenőrizd a Joinery log-ot:

```bash
sudo journalctl -u spaceos-joinery -n 100 --no-pager | grep -E "error|exception|500|saga" -i
```

Futtasd le a unit teszteket hogy kizárd kódhibát:
```bash
cd /opt/spaceos/spaceos-modules-joinery
dotnet test --filter "SubmitDoor"
```

### 2. Lehetséges okok

**A) CuttingProviderStub DI hiba** — a stub regisztrálva van, de az ICuttingProvider
injection valamilyen startup sorrendben hibát okoz. Ellenőrizd:
```csharp
// DependencyInjection.cs
services.AddScoped<ICuttingProvider, CuttingProviderStub>();
// ← Helyes? Vagy AddSingleton/AddTransient kellene?
```

**B) Graceful degradation try/catch hatókör** — a try/catch csak a
`SubmitCuttingSheetAsync` hívást védi, de esetleg más CuttingProvider-rel
kapcsolatos kód is dobhat és nincs védve.

**C) Adatállapot probléma** — a 29-joinery-order után az adatbázisban maradt
valamilyen inkonzisztens állapot, ami a következő saga-t blokkolja.

### 3. Fix

A hibát azonosítsd, javítsd, majd:

```bash
dotnet build   → 0 error
dotnet test    → 214/214 zöld
```

### 4. Ha adatállapot probléma (nem kódhiba)

Ha a 43-as teszt azért kapott 500-at mert a 29-es teszt után maradt valamilyen
dirty state az adatbázisban, azt nem kell itt javítani — helyette az E2E tesztnek
kell cleanup-ot végezni. Ebben az esetben írj DONE-t a diagnózis eredményével,
és a root az E2E terminálnak adja ki a cleanup fix-et.
