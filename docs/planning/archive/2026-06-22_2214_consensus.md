# SpaceOS Konsenzus Implementációs Terv

## Összefoglalás

Hibrid megközelítés: Phase 1-ben REST-alapú, cache-elt backend biztosítja a gyors szállítást és kockázat-mitigációt, de a frontend architektúra már WebSocket-ready. Assembly Planning unified timeline-nal (Planner-B UX koncepciója) indul REST-tel, Catalog Diff interactive slider-rel (Planner-B) JSON diff alapon (Planner-A library). Phase 2-re upgrade path WebSocket-re és prediktív ML-re, amikor már van baseline és mért használati adatok.

## Elfogadott prioritás sorrend

1. **Assembly Planning — Unified Timeline** (TOP 3 enhanced) — 5-6 nap
   - Legnagyobb üzleti érték (real-time gyártási láthatóság)
   - Planner-B UX koncepciója + Planner-A kockázat-kezelése
   - REST alapú, de WebSocket-ready architektúra

2. **Catalog Version Time-Travel** (TOP 2 enhanced) — 4-5 nap
   - Izolált fejlesztés, azonnali compliance érték
   - Planner-B slider UX + Planner-A diff library stack
   - Nincs külső függősége

3. **TOP 1 elvetése** — Mindkét tervező egyetért (redundáns TOP 3-mal)

## Backend szükségletek (összesített)

### Assembly Planning

**Phase 1 endpoint (REST + cache):**
```
GET /api/assembly/:id/timeline?merged=true
Response: {
  events: [
    {timestamp, type: "plan|actual", step, material, status: "ok|delay|ahead", 
     deltaMinutes: -15, plannedQty, actualQty}
  ],
  summary: {totalSteps, onTime, delayed, ahead, criticalPath}
}
```

**Implementáció:**
- Materialized view: `assembly_timeline_cache` (5 perc TTL)
- Query: JOIN `assemblies`, `bom_items`, `production_logs`
- Cache invalidation: production_logs INSERT trigger

**Phase 2 upgrade path:**
```
WebSocket: /ws/assembly/:id/live-stream
(SignalR/Socket.IO, auth middleware, reconnect logic)
```

### Catalog Version Management

**Endpoint-ok:**
```
GET /api/catalog/:id/timeline
Response: {
  snapshots: [{version, timestamp, authorId, changeCount, summary}],
  keyChanges: [{timestamp, field, impact: "price|material|dimensions"}]
}

GET /api/catalog/:id/version/:versionId
Response: {snapshot: {...}, diff: {changed, added, removed}}
```

**Adatmodell:**
- `product_versions` tábla: `(product_id, version, snapshot_json, created_at, user_id, change_summary)`
- Migration script: retrospektív snapshot generálás meglévő product-okból (ha nincs verziókezelés)

**Library:**
- Backend: `jsondiffpatch` (npm)
- Frontend: `react-diff-viewer` vagy custom timeline renderer

## Frontend megközelítés (legjobb elemek)

**Komponens struktúra:**
```

  ├─ LiveAssemblyTimeline (Planner-B koncepció, REST alapú)
  │   ├─ TimelineBubble (plan=outline, actual=solid overlay)
  │   ├─ DeltaBadge (reusable ±indikátor, színkódolt)
  │   └─ FollowModeToggle (auto-scroll mai napra)
  ├─ VarianceSummaryCard (top banner KPI-k)
  └─ FilterToolbar (date range, "deviations only")
```

**State management:**
```javascript
const useLiveAssembly = (assemblyId) => {
  // Phase 1: React Query (polling 30s)
```

**UX flow:**

- Plan events = szürke outline buborékok

**Mobil:** Egyetlen oszlop, pinch-to-zoom timeline skálázás, bottom sheet filter

## Technológiai kompromisszum

### Phase 1 (4 hét):
- REST endpoint + materialized view cache
- React Query polling (30s interval)
- Teljes UX (unified timeline, slider) működik

### Phase 2 (későbbi, ha Phase 1 sikeres):
- WebSocket infrastruktúra (SignalR setup, 1 hét)
- `useLiveAssembly` hook upgrade (drop-in replacement)
- Prediktív ML alert (training data gyűjtése Phase 1-ben)

**Indok:** Planner-A-nak igaza van, hogy WebSocket "1-2 nap" naiv (auth, scaling, reconnect = 1 hét). Planner-B-nek igaza van, hogy real-time az USP. **Megoldás:** architektúra támogatja a migrációt, de nem blokkolja a szállítást.
