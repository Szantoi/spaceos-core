# SpaceOS Konsenzus Implementációs Terv

## Összefoglalás

A konsenzus egy **pragmatikus, inkrementális MVP** stratégiát követ Planner-A sorrend logikájával, de integrál **B terv innovatív konfliktuskezelési** és újrafelhasználhatósági elemeit. A KPI Dashboard először (business anchor), majd Inline Edit (state pattern tanulás), végül egy **kibővíthető Filter rendszer** amely középutat képvisel a hardcoded és univerzális megközelítések között.

## Elfogadott prioritás sorrend

1. **KPI Dashboard (TOP 3)** — mindkét tervező egyetért: pszichológiai horgony, zero dependency, azonnali érték
2. **Inline Editing (TOP 2)** — row-level state management pattern tanulás, előkészíti a filter komplexitását
3. **RFQ Smart Filter (TOP 1)** — extendable architecture, nem full SQL DSL, de nem is hardcoded

## Backend szükségletek (összesített)

**Phase 1-2: NINCS** (localStorage elég)

**Phase 3 opcionális backend readiness:**
- `GET /api/rfqs?vendor=X&status=Y&date_from=Z` — query params alapú filter
- `PATCH /api/products/:id` — inline edit perzisztálás
- `GET /api/catalog/kpis` — cache-elt aggregációk (csak ha localStorage >5000 termék)

**Adatmodell bővítés (localStorage):**

## Frontend megközelítés (legjobb elemek)

### Phase 1: KPI Dashboard (2-3 óra)
**Komponensek:**
```
frontend/src/components/catalog/
  ├─ KPIDashboard.jsx (grid container, localStorage layout state)
  ├─ KPICard.jsx (metric + trend display)
  └─ hooks/useKPICalculator.js (memoized computations)
```

**Functionality:**
- **A terv:** statikus grid, gyors ship
- **B terv twist:** localStorage-ban mentett sorrend (nem drag-drop, csak config modal)
- **Trend tárolás:** B terv localStorage schema, de UI simplified (csak delta % megjelenítés)

### Phase 2: Inline Editing (4-5 óra)
**Komponensek:**
```
frontend/src/components/catalog/
  ├─ EditableCell.jsx (double-click to edit, Esc/Enter handling)
  ├─ hooks/useEditLock.js (localStorage conflict detection)
  └─ ConflictWarning.jsx (banner: "Another tab is editing this row")
```

**State management:**
- **A terv:** `editingCell` + `tempValue` state pattern
- **B terv twist:** `storage` event listener + timestamp-based lock check

**Conflict detection logic:**
```tsx
useEffect(() => {
  const handleStorageChange = (e) => {
    if (e.key === 'editLocks') {
      const locks = JSON.parse(e.newValue);
      if (locks[currentRowId] && locks[currentRowId].tabId !== currentTabId) {
        setConflictWarning(true);
      }
    }
  };
  window.addEventListener('storage', handleStorageChange);
}, [currentRowId]);
```

### Phase 3: RFQ Smart Filter (6-8 óra)
**Komponensek:**
```
frontend/src/components/shared/
  ├─ SmartFilter.jsx (generic filter container)
  ├─ FilterRow.jsx (field select, operator dropdown, value input)
  ├─ FilterPresets.jsx (saved queries dropdown)
  └─ hooks/useFilterState.js (URL sync + localStorage cache)
```

**Architecture kompromisszum:**
- **Nem SQL DSL** (B terv overengineering elkerülése)
- **De nem hardcoded** (A terv tech debt elkerülése)
- **Megoldás:** Config-driven filter rendszer

**Config példa:**
```tsx
const RFQ_FILTER_CONFIG = {
  fields: [
    {id: 'vendor', label: 'Supplier', type: 'multiselect', options: vendors},
    {id: 'status', label: 'Status', type: 'multiselect', options: STATUSES},
    {id: 'createdAt', label: 'Date', type: 'daterange'}
  ],
  operators: {
    multiselect: ['IN', 'NOT IN'],
    daterange: ['BETWEEN', '>', '<']
  }
};
```

**Újrafelhasználhatóság:**
```tsx
// RFQ view
<SmartFilter config={RFQ_FILTER_CONFIG} data={rfqs} onFilter={setFiltered} />

// Work Orders view (later)
<SmartFilter config={WORK_ORDER_FILTER_CONFIG} data={orders} onFilter={setFiltered} />
```

**URL sync (A terv):**
- `useEffect` hook szinkronizálja URL params ↔ filter state
- Single source of truth: URL a master

**Presets (B terv):**
- localStorage-ban mentett gyakori szűrések
- "Last 30 days", "High-value RFQs" gyors gombok

## Amit Sonnet-A-tól veszünk át

1. **Prioritás sorrend logika** — technikai függőség optimalizált (inline edit state pattern → filter state)
2. **URL-first state management** — `parseURLParams()` tiszta architektúra
3. **Optimistic update** stratégia — jobb UX mint conflict resolution modal
4. **Reális időbecslés** — 18-22h nettó dev, nem optimista 44h
5. **`react-window` performance mention** — előrelátó optimalizáció

## Amit Sonnet-B-től veszünk át

2. **KPI trend tárolás** — történeti összehasonlítás dashboard-on
3. **FilterPresets komponens** — mentett szűrések UX win
4. **Extendable filter architecture gondolkodás** — nem generic SQL, de nem is hardcoded
5. **"Ask Nexus" vízió** — Phase 4 roadmap item, nem most implementálva

## Nyitott kérdések a Conductor-nak

1. **Bundle size policy:** `react-grid-layout` (500KB) elfogadható KPI drag-drop-hoz, vagy CSS Grid elég? **Javaslat:** CSS Grid először, drag-drop Phase 4.

2. **Nexus AI integráció prioritás:** B terv "Ask Nexus" gombot javasol KPI kártyákra. Backend ready? **Javaslat:** Phase 4, spike ticket előtte.

3. **Multi-user környezet:** localStorage conflict detection **ideiglenes** megoldás. Mikor kezdjük WebSocket infrastructure-t? **Javaslat:** Spike ticket Q3 2026.

4. **Filter config centralizálás:** `FILTER_CONFIGS` registry pattern szükséges más view-khoz? **Javaslat:** Inline Edit után decision point, ne prematurely abstraháljunk.

5. **Performance baseline:** Milyen termékszámnál kell `react-window`? **Javaslat:** 1000+ termék vagy 100+ RFQ threshold, mérjünk Phase 1 után.

6. **Edit history tracking:** B terv említi a localStorage changeLog-ot. Audit követelmény? **Javaslat:** Nice-to-have, Phase 4 vagy később.
