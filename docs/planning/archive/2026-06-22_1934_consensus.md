# SpaceOS Konsenzus Implementációs Terv

## Összefoglalás

A konsenzus hibrid stratégiát követ: **EHS offline-first gyors win** (2-4 óra, 0 backend függőség) építi a felhasználói bizalmat, majd **Assembly variance backend-lean** megközelítéssel (polling kezdetben, WebSocket upgrade path), végül **Catalog diff** dekoratív bővítés. A terv egyesíti Planner-B értékalapú prioritását és Planner-A kockázatkezelő pragmatizmusát.

---

## Elfogadott prioritás sorrend

1. **EHS Incident Report** — offline-first, azonnali compliance érték, 0 backend blocker
2. **Assembly Variance Detection** — meglévő modul kiterjesztés, polling-first + WebSocket upgrade path
3. **Catalog Diff View** — tiszta UI munka, semantic diff megjelenítés

**Indoklás:** EHS shipelhető ma, bizalmat épít. Assembly second, mert 80% infrastructure kész van (WorkOrder modul 4.5). Catalog utoljára, mert ritkábban használt feature.

## Backend szükségletek (összesített)

### Sprint 1 (EHS)
```
POST /api/ehs/incidents          // v2 - később
POST /api/ehs/incidents/upload   // File storage endpoint
```
**Adatmodell** (`src/SpaceOS.Domain/Safety/Incident.cs`):
```csharp
public class SafetyIncident {
    public Guid Id { get; set; }
    public IncidentCategory Category { get; set; }
    public SeverityLevel Severity { get; set; }
    public string Location { get; set; }
    public GeoCoordinates? GPSLocation { get; set; }
    public string[] PhotoUrls { get; set; }
    public Guid ReportedBy { get; set; }
    public DateTime ReportedAt { get; set; }
    public bool IsSynced { get; set; } // Offline queue flag
}
```

### Sprint 2 (Assembly)
```
GET  /api/work-orders/{id}/variances
POST /api/work-orders/{id}/variances/{id}/approve
WS   /ws/assembly/{id}  // Optional upgrade path
```
**Adatmodell** - Event Sourcing approach:
```csharp
public class VarianceEvent {
    public Guid WorkOrderId { get; set; }
    public string BOMLineId { get; set; }
    public VarianceType Type { get; set; } // Quantity, Time, Material
    public decimal PlannedValue { get; set; }
    public decimal ActualValue { get; set; }
    public DateTime Timestamp { get; set; }
    public Guid ScannedBy { get; set; }
}
```

### Sprint 3 (Catalog)
```
GET /api/catalog/items/{id}/versions
GET /api/catalog/items/{id}/diff?from=v1&to=v2
```
**Response format** - Semantic diff:
```json
{
  "priceChange": { "old": 1200, "new": 1320, "percentChange": 10, "impact": "cost_increase" },

  "dimensionChanges": [...]
```

## Nyitott kérdések a Conductor-nak

### Architektúra döntések

1. **WebSocket infrastruktúra:** Socket.io vs SignalR vs native WebSocket? Assembly real-time push v2-ben szükséges?
2. **File storage:** EHS fotók tárolása blob storage (Azure) vagy local filesystem? Max file size limit?
3. **Offline storage kvóta:** IndexedDB 50MB elég EHS queue-hoz? Mi történik quota exceeded esetén?

### Üzleti prioritás validálás

4. **EHS compliance:** Van-e jogi követelmény a 24h-on belüli sync-re? Határozza meg a retry policy-t.
5. **Assembly variance threshold:** Melyik % felett kell automatikus alert? (Planner-B 10%-ot javasol)
6. **Catalog diff scope:** Csak price/leadtime, vagy dimension/material changes is kell?

### Performance és skálázás

7. **Assembly polling frekvencia:** 30s elfogadható gyártásban? Van-e SLA a variance detection-re?
8. **Catalog item limit:** 1000+ item esetén backend-side diff calculation szükséges? (Frontend only jelenleg)

### Security

9. **EHS GPS adatok:** GDPR compliance - tárolható munkavállalói helyzet?
10. **Variance approval:** Ki approve-olhat? Role-based access control kész a backend-en?

---

**Döntési javaslat:** EHS shipelhető ma (offline-only), Assembly sprint week 2 polling-gal indul (WebSocket decision pending), Catalog week 3 frontend-only (backend diff later ha scaling issue van).
