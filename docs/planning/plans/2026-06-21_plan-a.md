# Plan A — Független Implementációs Terv (Inkrementális, Biztonságos)

**Készítette:** Planner-A  
**Készült:** 2026-06-21  
**Megközelítés:** Inkrementális, walking skeleton, backend-first

---

## 1. Prioritás Sorrend

### P1 — Joinery API teljes integráció (1-2 sprint)
A legalacsonyabb kockázat. **Backend már létezik**, csak frontend hiányzik.

**Miért első:** Azonnali üzleti érték (Doorstar már LIVE), meglévő test coverage, nincs backend függőség.

### P2 — Offline audit cache (2-3 sprint)
**Miért második:** Gyártócsarnok realitás — internet szakadás nem állíthatja meg a munkát. Service Worker + IndexedDB készíti elő a PWA transzformációt.

### P3 — Workstation onboarding flow (1 sprint)
**Miért harmadik:** Új gyárak gyors beléptetése. QR kód + PIN státusz lekérdezés már van (Orchestrator), csak UX hiányzik.

### P4 — Floor mobile-first UX (3-4 sprint)
**Miért utolsó:** P2 offline réteg nélkül értelmetlen. Műhelyben használható design system szükséges.

---

## 2. Backend Szükségletek

### P1 — Joinery API (meglévő)
- ✅ `GET /joinery/api/designs` — Lista létező
- ✅ `POST /joinery/api/doors` — Create endpoint kész
- ✅ `GET /joinery/api/doors/{id}/bom` — Anyaglista

**Hiány:** Frontend routing + React komponensek

### P2 — Offline Audit Cache
**Új endpoint:**  
`POST /kernel/api/audit/buffer` — Offline gyűjtött audit események batch feltöltés

**Adatmodell:**
```csharp
public class OfflineAuditBuffer {
    public Guid DeviceId { get; set; }
    public List<AuditEvent> Events { get; set; }
    public DateTime ClientTimestamp { get; set; }
}
```

### P3 — Workstation Onboarding
**Meglévő használható:** `POST /orchestrator/api/workstations/pair`

**Hiányzik:**  
`GET /orchestrator/api/workstations/{id}/status` — QR validálás + státusz lekérdezés

### P4 — Floor Mobile-first UX
**Nincs backend igény** — tisztán UI réteg. Meglévő API-kat használ mobil layouttal.

---

## 3. Frontend Megközelítés

### P1 — Joinery Integráció
**Komponensek:**
- `JoineryDesignList.tsx` — Grid view, search, filter
- `DoorEditor.tsx` — Form komponens (méret, anyag, markolat)
- `BillOfMaterials.tsx` — Anyaglista display komponens

**State:** React Query (cache + refetch), localStorage draft mentéshez.

**Flow:**  
Lista → Szerkesztés → Preview → Mentés → BOM generálás

### P2 — Offline Cache
**Technológia:** Service Worker + IndexedDB (Dexie.js)

**Komponensek:**
- `OfflineIndicator.tsx` — Státusz banner
- `useSyncQueue` hook — Feltöltési logika

**Strategy:** Stale-while-revalidate + background sync

### P3 — Onboarding Flow
**Komponensek:**
- `WorkstationPairWizard.tsx` — Multi-step form
- `QRScanner.tsx` — Camera használat (react-qr-reader)
- `PINInput.tsx` — 6 digit confirmation

**UX:** 3 lépés: QR scan → PIN → Confirmation

### P4 — Mobile Floor UI
**Design System:** Tailwind mobile breakpoints + touch-friendly buttons (min 44px)

**Komponensek:**
- `MobileNav.tsx` — Bottom navigation
- `TaskCard.tsx` — Swipe gesture támogatás
- `QuickActions.tsx` — FAB menü

## 4. Kockázatok

| Kockázat | Hatás | Mitigáció |
|---|---|---|
| **P2 — Service Worker komplexitás** | Medium | Dexie.js (battle-tested), Workbox tanulmányozás |
| **P3 — QR Scanner browser support** | Low | Progressive enhancement, fallback manual input |
| **P4 — Touch gesture perf** | Medium | React 19 concurrent mode, virtualizálás nagy listáknál |
| **P2 — Offline sync konfliktus** | High | Last-write-wins, server timestamp authority |

## 5. Sorrend és Függőségek

```
P1 (Joinery) ────┐
                  ├──> P4 (Mobile UI)
P2 (Offline) ────┤
                  └──> P3 (Onboarding) ─> P4

P3 függ P2-től (QR scan offline cache-szel együtt működik gyárban)
P4 függ P1+P2-től (minden meglévő funkció mobil layoutja)
```

**Kritikus út:** P1 → P2 → P3 → P4  
**Párhuzamosítható:** P1 + P2 egyszerre indítható (külön feature branch)
