# Implementációs Terv — Planner-B (Merész)

## Stratégia: End-to-End Value Chain First

A hagyományos inkrementális építkezés helyett egy **teljes értéklánc早期 prototípust** építünk ki, ami azonnal demonstrálható ügyfeleknek.

---

## Prioritás sorrend

### 1. **Joinery API Integration → Teljes megrendelés-gyártás loop** (1. hét)
**Miért először:** A Doorstar soft launch kulcsa az, hogy egy megrendelést végigvigyünk a rendszeren — ajánlattól a gyártási lapig. Ez a legértékesebb demo.

**Backend:**
- `/api/joinery/products/configurator` — paraméterezett ajtó/ablak konfigurátor
- `/api/joinery/bom/generate` — anyaglista generálás szabályokból
- `/api/joinery/work-order/create` — munkalap előállítás

**Frontend új komponensek:**
- `ProductConfiguratorWizard.tsx` — lépésenkénti konfigurátor (méretek → anyagok → fittingek)
- `BOMPreview.tsx` — anyaglista előnézet árral
- `WorkOrderCard.tsx` — gyártási lap PDF generálás gombbal

**Kockázat:** Komplex parametrikus szabályok — kezdjük egyszerű sablon alapú szabályokkal (hardcode 5 ajtótípus), később bővíthető.

---

### 2. **Real-time Shop Floor Feed** (2. hét)
**Miért merész:** Ahelyett hogy várunk minden feature-re, azonnal bekapcsoljuk a **WebSocket élő státuszfrissítést** — a műhelyben dolgozó látja real-time hogy melyik gép mit csinál.

**Backend:**
- `/api/machines/status` SSE endpoint (már létezik SSE infrastruktúra a Dashboardban!)
- `IMachineStatusProvider` interfész — mock implementáció gépek nélkül
- Event bus: `MachineStatusChanged` domain event

**Frontend:**
- `ShopFloorLiveView.tsx` — nagy kijelző mód (TV a csarnokban)
- `MachineStatusBadge.tsx` — zöld/sárga/piros LED-szerű státusz
- `useSSEMachineStatus.ts` — újrafelhasználja a `useSSE` hookot

**Kockázat:** Nincs még valódi gép integráció — de mock adattal is MŰKÖDIK és impresszív. Később OPC-UA/Modbus driverekkel helyettesítjük.

---

### 3. **Offline-First PWA Enhancement** (3. hét)
**Miért radikális:** A magyar asztalosok 40%-ának ROSSZ az internet a gyárban. Service Worker + IndexedDB cache-el garantáljuk hogy offline is MŰKÖDIK.

**Backend:**
- Endpoint header: `Cache-Control: max-age=3600, stale-while-revalidate`
- `/api/sync/pending` — offline műveletek feltöltése online módba váltáskor

**Frontend:**
- `ServiceWorkerProvider.tsx` — Workbox integráció
- `OfflineIndicator.tsx` — sárga banner ha nincs net
- `PendingSyncBadge.tsx` — hány művelet vár szinkronra

**Kockázat:** Konfliktuskezelés (két ember szerkeszt offline ugyanazt a work order-t). Megoldás: **Last Write Wins + Conflict Log** — az admin látja és manuálisan feloldja.

## Technikai architektúra döntések

### State Management: **Zustand + TanStack Query**
- Zustand = lokál UI state (drawer open/closed, selected tab)
- TanStack Query = szerver state (cache, retry, optimistic updates)
- **NEM Redux** — túlkomplikált ennyi feature-höz

### API Layer: **tRPC helyett REST + Zod validáció**
- Backend OpenAPI spec (`/swagger`)
- Frontend: Fetch wrapper + Zod schema validáció
- Típusbiztonság: OpenAPI → TypeScript codegen

### Adatmodell evolúció:
```sql
-- Phase 1: Egyszerű joinery tables
CREATE TABLE joinery_products (id, name, params JSONB);
CREATE TABLE work_orders (id, product_id, status, machine_id);

-- Phase 2: Event sourcing készülék
CREATE TABLE machine_events (id, machine_id, event_type, payload JSONB, timestamp);
```
