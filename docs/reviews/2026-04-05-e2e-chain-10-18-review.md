## Priority 1 (Must Fix) -- CRITICAL

### 1. CRITICAL -- Silent test skips produce false green across ALL test files (10-18)

**Files**: All nine files
**Lines**: Every `it()` block containing `if (!chainWorks || ...) return;`

Every test in files 10 through 18 uses the pattern:

```ts
it('test name', async () => {
  if (!chainWorks || !someId) return;
  // actual test logic
});
```

When the guard returns early, vitest marks the test as **passed** -- not skipped. If `beforeAll` fails for any reason (network issue, server not running, transient 500), the entire suite silently passes with zero assertions executed. This is the single most damaging pattern in the test suite because CI will report green while testing nothing.

**Fix**: Use `it.skipIf()` or call `expect.assertions(N)` at the top of each test, or throw/skip explicitly:

```ts
it('test name', async () => {
  if (!chainWorks || !someId) {
    return expect(true).toBe(false); // or use it.skipIf
  }
  // ...
});
```

Alternatively, use vitest's `beforeAll` to call `describe.skip` or set a suite-level skip.

### 2. CRITICAL -- GDPR erase test accepts nearly any status code (17-gdpr.chain.test.ts, line 15)

```ts
expect([200, 400, 404, 422, 429]).toContain(res.status);
```

This test verifies the GDPR erase-user endpoint -- a legally mandated data deletion path. Accepting 400, 404, and 422 means the test passes even when the server rejects the request due to validation errors or the endpoint is missing. A broken GDPR erase flow would go undetected. For a compliance-critical endpoint, the test must assert that erasure actually succeeded (200) or fail loudly.

### 3. CRITICAL -- SSRF protection test accepts 404 as valid (15-nodes-sync.chain.test.ts, line 88)

```ts
expect([400, 404, 422, 429]).toContain(res.status);
```

The test labeled "rejects private IP (SSRF)" passes when the endpoint returns 404 (not deployed). This means the test will pass whether or not SSRF validation exists. A security test that passes when the security control is absent is worse than no test -- it provides false confidence.

**Fix**: Separate the "endpoint exists" check from the "SSRF is blocked" assertion:

```ts
if (res.status === 404) {
  console.warn('Endpoint not deployed -- SSRF test inconclusive');
  return; // or skip
}
expect([400, 422]).toContain(res.status);
```

### 4. CRITICAL -- Audit re-hash test silently passes with zero assertions (18-audit-rehash.chain.test.ts, lines 11-12)

```ts
if (tenants.status !== 200) return;
// ...
if (items.length === 0) return;
```

Two early returns with no skip indication. If the tenants endpoint fails or returns an empty list, the test passes with zero assertions. Combined with the broad `expect([200, 400, 404, 422, 429])` on line 24, this test provides almost no signal about whether audit re-hashing works.

## Priority 2 (Should Fix) -- HIGH

### 5. HIGH -- Overly broad status code assertions mask real failures (13-flowepic-full.chain.test.ts, line 62)

```ts
// delegate test
expect([200, 400, 409, 422, 429]).toContain(res.status);
```

Five accepted status codes. A 400 (malformed body) is a fundamentally different failure than a 200 (success). This test cannot distinguish between "delegation works" and "the request body format is wrong."

### 6. HIGH -- Proof upload test accepts 404 as valid (13-flowepic-full.chain.test.ts, line 84)

```ts
expect([201, 400, 404, 422, 429]).toContain(res.status);
```

A 404 means the BFF does not proxy multipart uploads at all. Accepting this means the test passes even when the feature is completely broken.

### 7. HIGH -- No cleanup of created test data (all files 10-18)

None of the test files implement `afterAll` cleanup. Each run of the suite creates:
- Multiple tenants (files 10, 11, 12, 13, 15, 16)
- Facilities, work-stations, space-layers, flow-epics
- Node registrations

Over time this pollutes the test environment with orphaned data, can cause name collisions, and may degrade query performance. More critically, leftover node registrations (file 15) could affect production federation behavior if tests run against a shared environment.

### 8. HIGH -- Token cache is unbounded and never invalidated (helpers.ts, line 8)

```ts
const tokenCache: Record<string, string> = {};
```

The token cache lives for the entire process lifetime. If a JWT expires mid-suite (e.g., long-running tests), all subsequent requests will use a stale token and get 401 errors. The tests broadly accept 429 but do not handle 401, so these failures would surface as unexpected status codes rather than clear auth failures.

### 9. HIGH -- Nodes register test asserts on a tautology (15-nodes-sync.chain.test.ts, line 33)

```ts
const body = res.data as { tenantId: string; serverUrl: string };
expect(body.tenantId || body).toBeDefined();
```

`body.tenantId || body` will always be truthy (if `tenantId` is falsy, `body` itself is the fallback, and `body` is always defined since it was just assigned). This assertion can never fail.

### 10. HIGH -- Sync signal test uses hardcoded fake UUID (15-nodes-sync.chain.test.ts, line 68)

```ts
epicId: '00000000-0000-0000-0000-000000000001', // fake epic
```

This tests sync signal with a non-existent epic, meaning the test only verifies the endpoint does not crash -- not that sync actually works. Combined with accepting 404 and 422, this test cannot verify any real sync behavior.

### 11. HIGH -- Chat test does not validate LLM output safety (14-chat.chain.test.ts)

The chat tests send `'Hello'` and `'List all tenants'` but never verify:
- Output does not contain system prompt leakage
- Output does not contain PII or internal identifiers
- Tool usage is scoped appropriately

For an LLM-integrated endpoint, the test should at minimum verify that `body.reply` does not contain known sensitive patterns (OWASP LLM06 -- Information Disclosure).

### 12. HIGH -- Flow epic delegate test does not verify the start succeeded (13-flowepic-full.chain.test.ts, line 56)

```ts
await PUT(`/bff/api/flow-epics/${epicId}/start`);
```

The result of `start` is discarded. If starting the epic fails (400, 422, 500), the delegate test proceeds against an epic in the wrong state, and the broad `expect([200, 400, 409, 422, 429])` will absorb the resulting error silently.

## Priority 3 (Recommended) -- MEDIUM

### 13. MEDIUM -- Unsafe type casts hide runtime shape mismatches (all files)

Every file uses `as` casts to assert response shapes:

```ts
const body = res.data as { items: unknown[]; totalCount: number };
```

If the server returns a different shape (e.g., `{ data: [...] }` instead of `{ items: [...] }`), the cast silences the type system and `body.items` becomes `undefined`. The subsequent `expect(body.items).toBeDefined()` would then fail with a confusing error instead of a clear schema mismatch message. Consider using zod or a runtime validator.

### 14. MEDIUM -- Snapshot test assumes starting an epic creates a snapshot (16-snapshots.chain.test.ts, line 30)

```ts
await PUT(`/bff/api/flow-epics/${epicId}/start`);
```

The result is not checked. If starting the epic does not produce a snapshot (or the start fails), both snapshot tests silently skip via the `if (!epicId) return` guard. Even if `epicId` is set, the snapshot may not exist yet, and the test accepts 404 as valid -- so there is no scenario where this test can fail.

### 15. MEDIUM -- Test ordering dependency not enforced (11-workstation-full.chain.test.ts)

The "reassign to different facility" test (line 50) depends on `workStationId` being created successfully, and implicitly depends on the "rename" test not corrupting the work station state. Vitest does not guarantee `it()` execution order by default unless `sequence` is configured.

### 16. MEDIUM -- `beforeAll` setup failures are swallowed silently (all files)

The `beforeAll` blocks use conditional returns (`if (tRes.status !== 201) return`) but never log why setup failed. When debugging CI failures, there is no way to know whether tenants, facilities, or sub-entities failed to create. At minimum, add `console.warn` on setup failure paths.

### 17. MEDIUM -- Chat "with tenant context" test sends empty context (14-chat.chain.test.ts, line 30)

```ts
context: {},
```

An empty context object does not actually test tenant-scoped chat behavior. This should include a real `tenantId` to verify the LLM correctly scopes its tool calls.

### 18. MEDIUM -- SpaceLayer intent test does not verify the update took effect (12-spacelayer-intent.chain.test.ts, line 39)

```ts
expect([200, 429]).toContain(res.status);
```

When the PUT returns 200, there is no follow-up GET to verify the intent data was actually persisted. Compare with file 11 (workstation rename) and file 13 (epic rename), which both do a verification GET after a successful update. This inconsistency suggests the verification was overlooked.

### 19. MEDIUM -- GDPR 401 test uses hardcoded localhost URL (17-gdpr.chain.test.ts, line 19)

```ts
const BFF = process.env.BFF_BASE_URL ?? 'http://127.0.0.1:3000';
```

This duplicates the same constant from `helpers.ts`. If the fallback URL changes, these inline copies will diverge. The `BFF` constant or a `rawFetch` helper should be exported from helpers.

## Priority 4 (Nice to Have) -- LOW

### 20. LOW -- File 13 proof upload bypasses the `bff()` helper (13-flowepic-full.chain.test.ts, lines 69-81)

The proof upload test uses raw `fetch()` and manually calls `getToken('admin')`, duplicating auth logic from the helper. If the auth flow changes (e.g., token format, header name), this test will break independently. Consider adding multipart support to the `bff()` helper.

### 21. LOW -- Inconsistent `beforeAll` guard patterns across files

Files 10, 11, 13 use `if (tRes.status === 201) tenantId = ...` (conditional assignment).
File 12 uses `if (tRes.status !== 201) return` (early return).
File 18 uses `if (tenants.status !== 200) return` inside the test itself rather than `beforeAll`.

This inconsistency makes it harder to audit the suite for silent-skip bugs.

### 22. LOW -- `uniqueName` collision risk is non-zero (helpers.ts, line 80)

```ts
return `${prefix}-${Date.now()}-${Math.random().toString(36).slice(2, 6)}`;
```

The random suffix is only 4 characters (36^4 = ~1.7M possibilities). For parallel CI runs sharing a database, this could collide. A UUID suffix would be safer.

### 23. LOW -- File 14 imports `getToken` but the main test bodies do not use it (14-chat.chain.test.ts, line 5)

```ts
import { POST, bff, getToken } from '../helpers';
```

`POST` is imported but never used; `getToken` is imported but only used implicitly through `bff()`. The unused imports are harmless but add noise.
